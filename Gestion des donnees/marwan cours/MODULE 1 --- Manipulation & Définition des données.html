1 Qu'est-ce que SQL ? — Détaillé

Définition simple :

SQL (Structured Query Language) est le langage standard utilisé pour communiquer avec une base de données relationnelle. Il sert à créer la structure, manipuler les données, interroger et contrôler les droits.

Pourquoi on l'utilise :

  • Pour stocker et retrouver des données rapidement.
  • Pour automatiser des tâches sur les données (insert/update/delete).
  • Pour structurer les données (tables, colonnes, contraintes).

Familles de commandes (avec rôle concret) :

  • DDL (Data Definition Language) — définir la structure : CREATE, ALTER, DROP.
    Ex. : créer une table clients.
  • DML (Data Manipulation Language) — modifier le contenu : INSERT, UPDATE, DELETE.
    Ex. : ajouter un client.
  • DQL (Data Query Language) — interroger : SELECT.
    Ex. : lister tous les clients d'Agadir.
  • DCL (Data Control Language) — contrôler les accès : GRANT, REVOKE.
    Ex. : donner le droit de lecture à un utilisateur.

2 Créer et supprimer une base de données (DDL) — détaillé

CREATE DATABASE nom;

Ce que ça fait : crée un espace logique qui contiendra des tables.

Exemple :

CREATE DATABASE eshop_app_db;
À savoir :
  • Après création, il faut USE eshop_app_db; pour travailler dedans.
  • Une base peut contenir plusieurs tables, vues, procédures.

DROP DATABASE nom;

Ce que ça fait : supprime la base ET toutes ses tables — irréversible si pas de sauvegarde.

Exemple :

DROP DATABASE eshop_app_db;
Piège : vérifier toujours avant d'exécuter (ne pas être connecté sur la base en production).

3 Créer une table — détail complet (types, contraintes)

Syntaxe de base :

CREATE TABLE clients (
  id INT NOT NULL AUTO_INCREMENT,
  nom VARCHAR(50) NOT NULL,
  prenom VARCHAR(40),
  date_naissance DATE,
  adresse VARCHAR(50) DEFAULT 'Agadir',
  PRIMARY KEY (id)
) ENGINE=InnoDB;

Explications ligne par ligne :

  • id INT : colonne numérique.
  • NOT NULL : la colonne ne peut pas être vide.
  • AUTO_INCREMENT : MySQL génère automatiquement la prochaine valeur (utile pour l'ID).
  • VARCHAR(50) : chaîne de caractères variable jusqu'à 50 caractères.
  • DEFAULT 'Agadir' : valeur utilisée si on ne fournit rien à l'insert.
  • PRIMARY KEY (id) : identifiant unique ; indexé automatiquement.
  • ENGINE=InnoDB : moteur de stockage (InnoDB supporte transactions, contraintes, verrous).

Types de données importants :

  • INT, BIGINT — entiers.
  • VARCHAR(n) — texte court.
  • TEXT — texte long.
  • DATE, DATETIME, TIMESTAMP — dates/temps.
  • DECIMAL(p,s) — nombres décimaux (argent).
  • BOOLEAN — vrai/faux (souvent stocké comme TINYINT(1)).

Contraintes utiles :

  • UNIQUE : interdit les doublons sur une colonne.
  • FOREIGN KEY : référence à une autre table (garantit l'intégrité référentielle).
  • CHECK (selon version) : condition sur la valeur.
Bonnes pratiques :
  • Définir des types justes (ni trop petits, ni trop gros).
  • Toujours ajouter une clé primaire.
  • Documenter la table.

4 Modifier une table (ALTER TABLE) — détails

Actions courantes :

Ajouter une colonne

ALTER TABLE clients ADD email VARCHAR(255);

Modifier un type

ALTER TABLE clients MODIFY COLUMN date_naissance DATE;

Changer le nom/structure

ALTER TABLE clients CHANGE COLUMN adresse ville VARCHAR(100);

Supprimer une colonne

ALTER TABLE clients DROP COLUMN email;

Renommer la table

ALTER TABLE commandes RENAME TO achats;
À savoir :
  • Certaines modifications lockent la table (peuvent rendre la table indisponible pendant l'opération) — planifier en heures creuses.
  • Faire une sauvegarde avant grosses modifications.

5 Supprimer une table (DROP TABLE) — détail 

DROP TABLE clients;

Ce que ça fait : efface structure + données.

Remarque : MySQL peut supprimer aussi les triggers liés ; vérifier dépendances.

6 SELECT — interrogation complète (clauses et ordre d'exécution)

Forme générale :

SELECT [DISTINCT] colonnes
FROM table1
[JOIN table2 ON condition]
[WHERE condition]
[GROUP BY champs]
[HAVING condition_groupe]
[ORDER BY champs]
[LIMIT offset, count];

Explication des clauses (avec ordre logique d'exécution interne) :

  1. FROM — on détermine les tables et les jointures.
  2. WHERE — filtre ligne par ligne (avant l'agrégation).
  3. GROUP BY — regroupe les lignes.
  4. HAVING — filtre sur les groupes (après agrégation).
  5. SELECT — calcule les colonnes demandées (agrégations, expressions).
  6. ORDER BY — tri final.
  7. LIMIT — nombre de lignes renvoyées.

Exemples :

Simple :

SELECT nom, prenom FROM clients WHERE adresse='Agadir' ORDER BY nom;

Avec agrégat :

SELECT filiere, AVG(moy1A) AS moyenne
FROM Stagiaires
GROUP BY filiere
HAVING AVG(moy1A) > 12;
Pièges :
  • WHERE ne peut pas filtrer sur un alias créé dans SELECT (utiliser sous-requête ou HAVING si agrégation).
  • Utiliser EXPLAIN pour comprendre pourquoi une requête est lente.

7 Les JOINTURES — expliqué en profondeur (ce que fait chacune)

Les jointures permettent de combiner des lignes venant de plusieurs tables selon une condition.

Imagine 2 tables :
clients(id, nom)
commandes(id, client_id, montant)

a) INNER JOIN (jointure interne)

Ce que fait : retourne les lignes où la condition de jointure est vraie dans les deux tables.

SELECT c.nom, o.montant
FROM clients c
INNER JOIN commandes o ON c.id = o.client_id;

Usage : quand on veut uniquement les clients qui ont des commandes.

b) LEFT (OUTER) JOIN (jointure gauche)

Ce que fait : retourne toutes les lignes de la table de gauche (clients), et les données de droite (commandes) si elles existent ; sinon NULL.

SELECT c.nom, o.montant
FROM clients c
LEFT JOIN commandes o ON c.id = o.client_id;

Usage : lister tous les clients et, si présent, leur commande. Utile pour trouver les clients sans commande (regarder WHERE o.client_id IS NULL).

c) RIGHT (OUTER) JOIN (jointure droite)

Ce que fait : symétrique de LEFT JOIN — toutes les lignes de la table de droite et correspondances à gauche.

Peu utilisée si tu peux inverser l'ordre des tables (préférer LEFT JOIN pour clarté).

d) FULL OUTER JOIN

Ce que ferait : union des LEFT + RIGHT (toutes les lignes des deux tables).

Important : MySQL classique ne supporte pas FULL OUTER JOIN directement — on simule avec UNION de LEFT et RIGHT.

e) CROSS JOIN (produit cartésien)

Ce que fait : combine chaque ligne de la table A avec chaque ligne de la table B (multiplication).

SELECT * FROM clients CROSS JOIN produits;

Usage : rarement ; attention au nombre de lignes (explosion combinatoire).

Notes pratiques sur les jointures :
  • Toujours préciser la condition ON ; sinon tu risques un produit cartésien par erreur.
  • Lorsque tu fais des JOIN sur des colonnes indexées (ex : client_id), la requête est beaucoup plus rapide.
  • Préférer des noms de table abrégés (c, o) pour lisibilité sur requêtes longues.
  • Pour déboguer, commence par INNER JOIN puis adapte en LEFT JOIN si tu dois conserver l'ensemble d'une table.

8 Vues (Views) — expliqué

Définition : une vue est une requête stockée accessible comme une table virtuelle.

Exemple :

CREATE VIEW vw_clients_agadir AS
SELECT id, nom, prenom FROM clients WHERE UPPER(adresse) = UPPER('agadir');

Points importants :

  • Une vue simplifie l'accès à des requêtes fréquentes.
  • Certaines vues sont modifiables (insertion possible) si elles sont simples (pas d'agrégat, pas de GROUP BY, pas de DISTINCT, pas de JOIN complexes).
  • Les vues ne stockent pas les données (sauf vues matérialisées, non native MySQL) — elles recalculent la requête à l'appel.
Bonnes pratiques :
  • Utilise les vues pour masquer la complexité (queries réutilisées).
  • Ne pas abuser : les vues complexes peuvent cacher des problèmes de performance.

9 Tables temporaires — expliqué (usage et comportement)

Définition : table visible seulement dans ta session (connexion) et automatiquement supprimée à la fin de la session.

Exemple :

CREATE TEMPORARY TABLE temp_clients
SELECT id, nom FROM clients WHERE adresse='Agadir';

Utilisations courantes :

  • Stocker un résultat intermédiaire pour un traitement lourd.
  • Améliorer lisibilité : découper une grosse requête en étapes.
  • Éviter recalculs coûteux dans une même session.
Remarques :
  • La table temporaire masque une table normale du même nom dans la session.
  • Elle est isolée : d'autres utilisateurs ne la voient pas.
  • Pas pour stockage permanent (utiliser table normale si besoin permanent).

10 INSERT / UPDATE / DELETE — détails pratiques

INSERT

Une ligne

INSERT INTO clients (nom, prenom, date_naissance)
VALUES ('Zaki', 'Fatima', '1986-05-22');

Plusieurs lignes

INSERT INTO produits (nom, prix) VALUES ('Clavier', 150), ('Souris', 80);
Remarques :
  • Si une colonne a DEFAULT, tu peux l'omettre dans la liste de colonnes.
  • Si AUTO_INCREMENT, tu peux ignorer la colonne id.

UPDATE

UPDATE clients
SET adresse = 'Rabat', prenom = 'Fadia'
WHERE id = 2;
ATTENTION : si tu oublies le WHERE, tu mettras à jour toutes les lignes ! Toujours vérifier avec SELECT avant un UPDATE massif.

DELETE

DELETE FROM clients WHERE id = 3;
ATTENTION : DELETE FROM clients; supprime tout.
Bonnes pratiques :
  • Faire un SELECT avec la même WHERE avant DELETE ou UPDATE pour vérifier l'impact.
  • Utiliser transaction (BEGIN / COMMIT / ROLLBACK) pour réversibilité (voir module transactions).

11 Autres notions utiles (débutant) — récapitulatif rapide

LIMIT — pagination / limiter les résultats

SELECT * FROM produits LIMIT 10 OFFSET 20;

DISTINCT — valeurs uniques

SELECT DISTINCT option FROM Stagiaires;

ORDER BY — tri

SELECT nom, prenom FROM clients ORDER BY nom ASC, prenom DESC;

GROUP BY + agrégats (intro)

SELECT filiere, COUNT(*) AS nb
FROM Stagiaires
GROUP BY filiere;

COUNT, SUM, AVG, MIN, MAX — fonctions d'agrégat.

12 Erreurs fréquentes des débutants (et comment les éviter)

  • Oublier le WHERE dans UPDATE/DELETE → toujours vérifier par un SELECT.
  • Utiliser SELECT * en production → choisir les colonnes nécessaires.
  • Mettre % au début d'un LIKE (LIKE '%abc') → empêche l'utilisation d'index.
  • Jointure sans ON → produit cartésien (résultat explosif).
  • Mauvais type de colonne (ex : stocker un prix en VARCHAR) → problèmes de calcul.
  • Pas d'index sur colonnes de jointure → requêtes très lentes.
