Débutant 🟦 PARTIE A — NIVEAU DÉBUTANT

⭐ 1. Qu’est-ce que la sécurité d’une base de données ?

La sécurité dans MySQL protège :

  1. 1️⃣ Les données
  2. 2️⃣ Le serveur MySQL lui-même
  3. 3️⃣ Les utilisateurs qui y accèdent
  4. 4️⃣ Le réseau qui transporte les données

Objectifs :

  • ✔ Confidentialité
  • ✔ Intégrité
  • ✔ Disponibilité
⭐ 2. Les 9 bonnes pratiques essentielles (OFPPT)

Selon ton document OFPPT :

  • ✔ Sécuriser physiquement les serveurs
  • ✔ Utiliser un serveur de base de données séparé
  • ✔ Configurer un proxy HTTPS
  • ✔ Activer un protocole de cryptage (SSL/TLS)
  • ✔ Faire des sauvegardes régulières
  • ✔ Mettre à jour les applications
  • ✔ Utiliser une authentification forte
  • ✔ Attribuer des rôles adaptés aux utilisateurs
  • ✔ Éviter les ports par défaut
⭐ 3. Sauvegarder une base de données

La commande la plus utilisée : mysqldump

✔ Sauvegarder une BDD complète :

Shell
mysqldump -u root -p nom_bd > sauvegarde.sql
⭐ 4. Restaurer une base

Étape 1 : créer la base

SQL
CREATE DATABASE nom_bd;

Étape 2 : restaurer

Shell
mysql -u root -p nom_bd < sauvegarde.sql
⭐ 5. Créer un utilisateur MySQL

Syntaxe officielle :

SQL
CREATE USER 'login'@'hote'
IDENTIFIED BY 'mot_de_passe';

Exemples :

SQL
CREATE USER 'saadi'@'localhost'
IDENTIFIED BY 'pass123';

CREATE USER 'bensaid'@'194.28.12.4'
IDENTIFIED BY 'bensaid99';

Utilisateur pouvant se connecter de partout :

SQL
CREATE USER 'hanane'@'%'
IDENTIFIED BY 'mdp123';
⭐ 6. Modifier / supprimer un utilisateur

Renommer :

SQL
RENAME USER 'ancien'@'localhost'
TO 'nouveau'@'localhost';

Changer le mot de passe :

SQL
SET PASSWORD FOR 'login'@'localhost'
    = PASSWORD('nouveau_mdp');

Supprimer :

SQL
DROP USER 'login'@'localhost';
⭐ 7. Accorder des privilèges

Syntaxe :

SQL
GRANT privilège
ON objet
TO utilisateur;

Exemple réel (documentation OFPPT) :

SQL
GRANT SELECT, INSERT, UPDATE, DELETE
ON eshop_app_db.clients
TO 'saadi'@'localhost';

Accorder SELECT sur toutes les bases :

SQL
GRANT SELECT ON *.*
TO 'saadi'@'localhost';
⭐ 8. Retirer un privilège
SQL
REVOKE INSERT
ON eshop_app_db.clients
FROM 'saadi'@'localhost';
⭐ 9. Voir les utilisateurs et leurs privilèges

Liste des utilisateurs :

SQL
SELECT user, host
FROM mysql.user;

Voir les privilèges d’un utilisateur :

SQL
SHOW GRANTS FOR 'saadi'@'localhost';

Expert 🟪 PARTIE B — NIVEAU EXPERT

⭐ 1. Comprendre les niveaux de privilèges

MySQL permet d’accorder des permissions à 4 niveaux :

Niveau Exemples Usage
Global *.* SELECT, ALL Administrateurs
Base nom_bd.* SELECT, INSERT Applications
Table nom_bd.table SELECT, UPDATE Contrôle fin
Routine EXECUTE Procédures/fonctions
⭐ 2. Liste des privilèges importants
  • ✔ SELECT / INSERT / UPDATE / DELETE
  • ✔ CREATE TABLE / ALTER / DROP
  • ✔ CREATE VIEW / CREATE ROUTINE / EXECUTE
  • ✔ LOCK TABLES
  • ✔ GRANT OPTION → permet d’attribuer des privilèges à d’autres (dangereux ⚠️)
⭐ 3. Les rôles MySQL (professionnels)

Ton document OFPPT introduit les rôles :

✔ Créer un rôle :

SQL
CREATE ROLE 'webdeveloper';

✔ Accorder des privilèges au rôle :

SQL
GRANT SELECT ON mysql.user
TO 'webdeveloper';

✔ Associer un utilisateur à un rôle :

SQL
GRANT 'webdeveloper'
TO 'rh_salmi'@'entreprise.com';

✔ Activer un rôle :

SQL
SET ROLE 'webdeveloper';

✔ Définir un rôle par défaut :

SQL
CREATE USER 'u2'@'%'
IDENTIFIED BY 'pass'
DEFAULT ROLE 'webdeveloper';
⭐ 4. Meilleures pratiques de sécurité (professionnelles)
  • ✔ 1. Ne jamais utiliser root pour une application
    Créer un utilisateur dédié :
    SQL
    CREATE USER 'app_user'@'localhost'
IDENTIFIED BY 'appPass2024';

GRANT SELECT, INSERT, UPDATE
ON shop_db.* TO 'app_user'@'localhost';
  • ✔ 2. Séparer les environnements
    dev / test / production (avec comptes très restreints)
  • ✔ 3. Interdire l’accès root via réseau
    Modifier my.cnf :
    Config
    skip-networking
bind-address = 127.0.0.1
  • ✔ 4. Activer SSL/TLS pour les connexions distantes
    Obligatoire dans les entreprises.
  • ✔ 5. Toujours faire une sauvegarde avant un changement critique
    Shell
    mysqldump -u root -p --all-databases > save_all.sql
⭐ 5. Architecture de sécurité idéale
  • Firewall : ports limités, MySQL pas accessible publiquement
  • Proxy HTTPS : cryptage des flux
  • MySQL : sécurisé, rôles, privilèges, configuration stricte
  • Application : validation des entrées, protection SQL injection
  • Logs + audit : triggers d’historisation, journaux de sécurité
⭐ 6. Exemple complet professionnel : rôle + utilisateur + privilèges
SQL
-- créer un rôle
CREATE ROLE 'gestion_clients';

-- donner des permissions au rôle
GRANT SELECT, INSERT, UPDATE
ON eshop_app_db.clients
TO 'gestion_clients';

-- créer un utilisateur
CREATE USER 'amine'@'%'
IDENTIFIED BY 'AmiN123$$';

-- attribuer le rôle
GRANT 'gestion_clients'
TO 'amine'@'%';

-- définir le rôle par défaut
SET DEFAULT ROLE 'gestion_clients'
TO 'amine'@'%';
⭐ 7. Exemple expert : gestion des accès lors d’une migration

Créer un rôle "lecture seule" :

SQL
CREATE ROLE 'lecture_seule';
GRANT SELECT ON *.* TO 'lecture_seule';

Assigner temporairement à des utilisateurs :

SQL
GRANT 'lecture_seule' TO 'dev1'@'%';
SET ROLE 'lecture_seule';

➜ Utile pour maintenance, audits ou migration.
