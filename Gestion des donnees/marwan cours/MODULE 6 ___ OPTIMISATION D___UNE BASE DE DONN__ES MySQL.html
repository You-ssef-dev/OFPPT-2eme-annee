Débutant 🟦 PARTIE A — NIVEAU DÉBUTANT

⭐ 1. Pourquoi optimiser MySQL ?

Parce que :

  • Un site lent fait fuir les utilisateurs
  • Une requête mal écrite peut multiplier les coûts par 100
  • Les performances chutent quand les données augmentent
  • Un système non optimisé consomme trop de RAM / CPU

MySQL peut aller TRÈS vite…
👉 mais uniquement si tu l’aides un peu.

⭐ 2. Les 9 règles d’or (simples mais puissantes)

D’après le document OFPPT :

  • ✔ 1. Éviter SELECT *
    Pourquoi ?
    → Charge inutile
    → Empêche l’utilisation de certains index
    → Rend les requêtes plus lourdes
    À faire :
SQL
SELECT id, nom, prix
FROM produit;
  • ✔ 2. Éviter WHERE … IN (...) si la liste est grande
    Préférer EXISTS, plus rapide :
SQL
SELECT *
FROM produit p
WHERE EXISTS (
    SELECT 1
    FROM categorie c
    WHERE c.id = p.id_cat
);
  • ✔ 3. Ne pas utiliser de fonctions dans WHERE
    Exemple lent :
SQL
WHERE UPPER(nom) = 'ALI'  -- lent

MySQL ne peut pas utiliser l’index.

À faire :

SQL
WHERE nom = 'Ali'  -- rapide
  • ✔ 4. Éviter les LIKE commençant par %
SQL
WHERE nom LIKE '%ali'   -- Très lent ❌

Car MySQL ne peut pas savoir où rechercher.
À faire :

SQL
WHERE nom LIKE 'Ali%'   -- rapide ✔
  • ✔ 5. Utiliser INNER JOIN plutôt que LEFT JOIN quand possible
    Les LEFT JOIN sont toujours plus lourds.
  • ✔ 6. Éviter ORDER BY inutile
    Le tri est très coûteux.
  • ✔ 7. Éviter DISTINCT sauf si indispensable
    Beaucoup pensent que DISTINCT sert “à nettoyer”. ❌
    → Il faut corriger la requête, pas masquer le problème.
  • ✔ 8. Attention au “problème N+1”
    Si tu fais une requête SELECT dans une boucle :
    ❌ 1 requête principale + 100 petites requêtes → tueur de performances.
  • ✔ 9. Utiliser les requêtes préparées
    Elles permettent à MySQL de garder un plan d’exécution en mémoire.
⭐ 3. Les requêtes préparées (PREPARE)

Extrait OFPPT :

SQL
PREPARE stmt1 FROM
'SELECT id, designation
 FROM produit
 WHERE id = ?';

SET @id = 999;
EXECUTE stmt1 USING @id;

DEALLOCATE PREPARE stmt1;

Avantages :

  • ✔ optimisation interne
  • ✔ protège contre les injections SQL
  • ✔ réduit la charge réseau
⭐ 4. Introduction aux index (concept simple)

Un index est comme l’index d’un livre :

  • → il permet à MySQL d’aller directement à l’endroit voulu
  • → au lieu de lire toutes les lignes

Sans index :
MySQL lit TOUTE la table (table scan) 😱

Avec index :
MySQL va directement au bon emplacement ✔

⭐ 5. Créer un index (simple)
SQL
CREATE INDEX idx_clients_adresse
ON clients(adresse);

Index unique :

SQL
CREATE UNIQUE INDEX idx_email
ON clients(email);
⭐ 6. Quand créer un index ?

Créer un index sur une colonne :

  • ✔ utilisée dans WHERE
  • ✔ utilisée dans ORDER BY
  • ✔ utilisée dans INNER JOIN
  • ✔ très filtrante (email, id, ville, etc.)

Ne pas indexer :

  • colonnes à faibles valeurs distinctes (H/F, oui/non)
  • colonnes rarement utilisées
  • champs TEXT très grands
⭐ 7. Comment vérifier si une requête utilise un index ?
SQL
EXPLAIN
SELECT *
FROM clients
WHERE adresse = 'Agadir';

Très important pour diagnostiquer les lenteurs.

Expert 🟪 PARTIE B — NIVEAU EXPERT

⭐ 1. Les différents types d’index

D’après ton document :

  • ✔ B-Tree Index
    → Le plus courant
    → Pour la majorité des requêtes
  • ✔ FULLTEXT Index
    → Pour la recherche de texte longue
    → Exemple : blog, produits, articles
  • ✔ SPATIAL Index
    → Données géographiques (cartes, GPS)
⭐ 2. Index clusterisé vs non clusterisé
  • ✔ L’index PRIMARY est clusterisé
    → organise physiquement les données selon la clé primaire
    → très rapide pour les recherches par ID
  • ✔ Les autres index sont secondaires
    → pointent vers les lignes réelles (double accès)
⭐ 3. Stratégie d’indexation professionnelle
  • ✔ Index sur clé primaire
  • ✔ Index sur colonnes utilisées dans WHERE
  • ✔ Index unique sur email / username
  • ✔ Index multi-colonnes quand la requête filtre sur plusieurs champs
  • ✔ Attention : trop d’index = INSERT/UPDATE/DELETE plus lents
⭐ 4. Index multi-colonnes (optimisation avancée)
SQL
CREATE INDEX idx_nom_prenom
ON clients(nom, prenom);

Règle d’or :
→ MySQL n’utilise l’index que de gauche à droite.

Cet index marche pour :

  • ✔ nom
  • ✔ nom + prenom
  • ❌ prenom seul
⭐ 5. Optimisation structurelle (modèle de données)
  • ✔ Normaliser jusqu’à 3NF
  • ✔ Éviter les champs inutiles
  • ✔ Utiliser les bons types :
    • INT vs BIGINT
    • VARCHAR(100) plutôt que VARCHAR(255)
    • DATE plutôt que VARCHAR pour une date
  • ✔ Séparer les données historiques dans des tables d’archives
⭐ 6. Optimisation du serveur MySQL

Extrait OFPPT :

  • ✔ Utiliser SSD plutôt que HDD
  • ✔ Ajuster la RAM (buffer pool InnoDB)
  • ✔ Utiliser InnoDB au lieu de MyISAM
  • ✔ Installer un outil d’analyse :
    • MySQLTuner
    • Tuning-primer
  • ✔ Garder MySQL à jour
⭐ 7. Améliorer les performances via le cache

MySQL met en cache :

  • les résultats de requêtes
  • les plans d’exécution
  • les index en mémoire

Tu dois :

  • ✔ limiter la taille des colonnes
  • ✔ éviter SELECT *
  • ✔ créer des index pertinents
⭐ 8. EXPLAIN — l’outil d’analyse le plus important
SQL
EXPLAIN
SELECT *
FROM produits
WHERE prix > 100;

À analyser :

  • type (ALL = mauvais, ref/index = bon)
  • key (index utilisé)
  • rows (nombre de lignes parcourues)
  • Extra (Using where, Using index, etc.)
⭐ 9. Exemple expert d’optimisation d’une requête

Requête de départ :

SQL
SELECT *
FROM produits
WHERE categorie = 'PC'
  AND prix > 1000
ORDER BY prix DESC;

Optimisation :

  • Index multi-colonne :
SQL
CREATE INDEX idx_cat_prix
ON produits(categorie, prix);
  • Ne sélectionner que les colonnes nécessaires :
SQL
SELECT id, nom, prix
FROM produits
WHERE categorie = 'PC'
  AND prix > 1000
ORDER BY prix DESC;

Filtrer avant de trier : MySQL trie seulement les lignes filtrées → beaucoup plus rapide.

⭐ 10. Optimisation des requêtes JOIN

Toujours indexer :

  • la clé étrangère
  • la colonne de jointure

Exemple :

SQL
SELECT p.nom, c.nom_cat
FROM produits p
INNER JOIN categorie c
    ON p.id_cat = c.id;

Index recommandés :

SQL
CREATE INDEX idx_p_idcat ON produits(id_cat);
CREATE INDEX idx_c_id    ON categorie(id);
