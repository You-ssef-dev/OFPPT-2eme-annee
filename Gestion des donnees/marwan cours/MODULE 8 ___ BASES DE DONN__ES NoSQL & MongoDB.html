=============================== 🟦 PARTIE A — INTRODUCTION NoSQL (Débutant) ===============================
🟦 PARTIE A — INTRODUCTION NoSQL
Niveau : Débutant
⭐ 1. Pourquoi le NoSQL ?

Le SQL a dominé pendant 40 ans, mais il présente des limites :

  • ❌Scalabilité verticale
  • ❌Données non structurées difficiles à stocker
  • ❌Taux d’écriture énorme difficile à absorber
  • ❌Modifications de schéma compliquées

Les applications modernes (réseaux sociaux, big data, e-commerce, IoT) ont besoin de :

  • ✔Scalabilité horizontale
  • ✔Flexibilité
  • ✔Rapidité
  • ✔Stockage massif

👉 C'est là que les bases NoSQL sont apparues.

⭐ 2. Types de bases NoSQL

Il existe 4 grandes familles :

1️⃣ Base clé-valeur (Key-Value)

Exemples : Redis, DynamoDB

  • ✔Ultra rapide
  • ✔Idéal pour sessions, tokens, cache

2️⃣ Base orientée documents (Document Store)

Exemples : MongoDB, CouchDB

  • ✔Stocke des documents JSON
  • ✔Très flexible
  • ✔Modélisation simple

3️⃣ Base orientée colonnes (Column Store)

Exemples : Cassandra, HBase

  • ✔Parfaite pour les Big Data
  • ✔Optimisée pour les écritures massives

4️⃣ Base orientée graphes (Graph DB)

Exemples : Neo4j

  • ✔Relations complexes
  • ✔Réseaux sociaux, recommandations

👉 MongoDB = Base orientée documents.

⭐ 3. JSON & BSON (format de stockage MongoDB)

MongoDB utilise JSON pour la présentation
et BSON pour le stockage interne (plus rapide, typé).

Exemple JSON :

{
  "nom": "Salma",
  "age": 22,
  "ville": "Agadir"
}

Types supportés :

  • ✔string
  • ✔integer
  • ✔double
  • ✔boolean
  • ✔object
  • ✔array
  • ✔null
  • ✔date
=============================== 🟦 PARTIE B — MODÉLISATION DOCUMENTAIRE (Intermédiaire) ===============================
🟦 PARTIE B — MODÉLISATION DOCUMENTAIRE
Niveau : Intermédiaire

MongoDB n’impose pas de schéma strict → tu choisis ta structure selon le besoin.

Deux stratégies :

⭐ 1. Embedding (imbriqué)
{
  "client": "Ali",
  "commandes": [
    { "ref": 1, "total": 800 },
    { "ref": 2, "total": 500 }
  ]
}
  • ✔rapide
  • ✔une seule lecture
  • ✔idéal pour relations 1-N
⭐ 2. Linking (référencement)
{
  "client": "Ali",
  "commande_id": 22
}
  • ✔documents plus légers
  • ✔meilleures mises à jour
  • ✔idéal pour N-N ou gros documents
⭐ 3. Bonnes pratiques de modélisation
  • ✔Mettre les données utilisées ensemble dans le même document
  • ✔Éviter les documents > 16 Mo (limite MongoDB)
  • ✔Utiliser l’embedding pour 1-N sans croissance importante
  • ✔Utiliser les références pour N-N ou données volumineuses

=============================== 🟦 PARTIE E — AGRÉGATION (Niveau Expert) ===============================
🟦 PARTIE E — AGRÉGATION
Niveau : Expert

MongoDB possède un pipeline d’agrégation très puissant.

Exemple : total des achats :

db.clients.aggregate([
  { $unwind: "$achats" },
  { $group: { _id: "$nom", total: { $sum: "$achats.prix" } } }
])

Étapes importantes :

  • $match: filtrer
  • $project: remodeler
  • $unwind: éclater les tableaux
  • $group: regrouper / calculs
  • $sort: trier
  • $limit: limiter

=============================== 🟦 PARTIE H — MongoDB + Python (PyMongo) ===============================
🟦 PARTIE H — MongoDB + Python (PyMongo)
Niveau : Intermédiaire → Expert
⭐ 1. Installer PyMongo
pip install pymongo
⭐ 2. Connexion Python → MongoDB
from pymongo import MongoClient

client = MongoClient("mongodb://localhost:27017/")
db = client["boutique"]
⭐ 3. Insérer en Python
db.clients.insert_one({"nom": "Ali", "ville": "Agadir"})
⭐ 4. Lire en Python
for c in db.clients.find({"ville": "Agadir"}):
    print(c)
⭐ 5. Agrégation Python
pipeline = [
    { "$unwind": "$achats" },
    { "$group": { "_id": "$nom", "total": { "$sum": "$achats.prix" } } }
]

for doc in db.clients.aggregate(pipeline):
    print(doc)
